# MASTER PROMPT ДЛЯ CLAUDE CODE: ПЕРЕСТРОЙКА MOTORA MVP В MOTORHOF CRM

Ты работаешь как lead full-stack engineer и product architect над уже существующим проектом CRM автосалона. Не создавай новый проект с нуля. Сначала внимательно изучи текущий репозиторий, используемый стек, структуру БД, маршруты, компоненты, авторизацию и существующие данные. Затем последовательно измени текущий проект согласно требованиям ниже.

## 1. Контекст проекта

Сейчас существует первая MVP под названием MOTORA. В ней есть:

- Dashboard;
- Fahrzeuge/Автомобили;
- Deals/Сделки;
- Kunden/Клиенты;
- Aufgaben/Задачи;
- упрощённые статусы склада;
- упрощённый расчёт закупки, расходов, себестоимости и маржи;
- тёмный интерфейс с оранжевым акцентом;
- отображение денежных сумм в долларах;
- российские названия источников и тестовые данные.

Проект нужно превратить в внутреннюю CRM австрийского автосалона MOTORHOF OG. Главным объектом системы является автомобиль. Все расходы, задачи, просмотры, документы, продажи, брони и гарантийные случаи должны быть связаны прежде всего с автомобилем.

CRM не является полноценной бухгалтерской программой и не заменяет Steuerberater, но её управленческие расчёты должны быть математически корректными, прозрачными и пригодными для проверки.

## 2. Правила выполнения работы

1. Не создавай новый репозиторий и не меняй технологический стек без объективной необходимости.
2. Не переписывай работающие части проекта только ради другого стиля кода.
3. Не удаляй существующие данные необратимо.
4. Перед изменениями изучи существующие модели и подготовь безопасную миграцию.
5. Если текущие названия моделей отличаются от названий в этом задании, адаптируй существующие модели, а не создавай дубликаты.
6. Все финансовые вычисления выполняй на сервере или в общем проверяемом domain/service layer. Клиентская часть может показывать live preview, но не является источником истины.
7. Для денег используй Decimal/NUMERIC с двумя знаками после запятой. Не используй JavaScript floating point как постоянный источник финансовых значений.
8. Все вычисляемые значения округляй до двух знаков по единому правилу.
9. Не хардкодь статусы, категории и роли в разных компонентах. Используй централизованные enum/config и единые labels.
10. Все критические проверки должны существовать и на frontend, и на backend.
11. Сохрани адаптивность интерфейса для desktop и tablet. Основной сценарий — desktop.
12. Не добавляй платные внешние сервисы без необходимости.
13. Не сохраняй загруженные файлы в локальную файловую систему Vercel. Используй уже существующее объектное хранилище. Если его нет, создай storage abstraction и явно задокументируй необходимую настройку.
14. Не заявляй, что функция готова, если она существует только визуально и не сохраняет данные.

## 7. Код учёта и парковочное место

Не хранить строку учётного номера как единственное значение. Хранить отдельно:

- `systemSequence` — автоматически создаваемый порядковый номер, неизменяемый;
- `parkingRow` — одна латинская буква A–Z, uppercase;
- `parkingSpot` — положительное целое число;
- `internalCode` — вычисляемое отображение.

Формат отображения:

```text
A-12 / MH-0042
```

Если автомобиль ещё в дороге и места нет:

```text
— / MH-0042
```

Правила:

- MH-номер никогда не меняется;
- парковочное место можно менять;
- одно активное парковочное место нельзя назначить двум непроданным автомобилям;
- после продажи место освобождается, но история остаётся;
- вести историю перемещений: старое место, новое место, дата, пользователь.

### 10.2 Налоговый режим `taxScheme`

## 12. Финансовые формулы

### 12.1 Общие правила

- Все цены продажи и минимальные цены в UI вводятся gross, в EUR.
- Все формулы должны быть вынесены в тестируемый financial calculation service.
- VAT rate хранить как настройку/поле, default для автомобиля — 20%.
- Налог не может быть отрицательным.
- Расходы автомобиля уменьшают экономическую маржу, но не должны автоматически уменьшать Einkaufspreis gemäß §24.
- У каждого расхода должна быть отметка, включён ли он уже в полную стоимость закупки, чтобы избежать двойного вычитания.

### 12.2 Differenzbesteuerung

Использовать:

```text
bruttoDifferenz = max(0, actualSalePriceGross - einkaufspreisGemaess24)

differenzUst = bruttoDifferenz * vatRate / (100 + vatRate)

при vatRate = 20:
differenzUst = bruttoDifferenz * 20 / 120
differenzUst = bruttoDifferenz / 6
```

Нельзя использовать:

```text
(Verkaufspreis - Einkaufspreis) * 0.2
```

Потому что Umsatzsteuer уже содержится в положительной brutto-разнице.

Маржа до дополнительных расходов:

```text
marginBeforeAdditionalExpenses =
  actualSalePriceGross
  - totalCashAcquisitionCost
  - differenzUst
```

Финальная маржа:

```text
finalVehicleMargin =
  marginBeforeAdditionalExpenses
  - sum(linkedVehicleExpenses not already included in totalCashAcquisitionCost)
```

Для Privat:

```text
totalCashAcquisitionCost = privatePurchasePrice
einkaufspreisGemaess24 = privatePurchasePrice по умолчанию
```

Для Auktion:

```text
totalCashAcquisitionCost = auctionInvoiceTotal + отдельные не включённые расходы закупки
einkaufspreisGemaess24 = подтверждённое значение из счёта/налоговых данных
```

Пример unit test:

```text
Privat purchase = 10 000
Sale gross = 12 000
Brutto difference = 2 000
Differenz-USt = 333.33
Margin before expenses = 1 666.67
```

Пример Auktion:

```text
Fahrzeugpreis = 10 000
Auktionsrechnung gesamt = 10 800
Einkaufspreis gemäß §24 = 10 000
Sale gross = 12 000
Differenz-USt = 333.33
Margin before additional expenses = 866.67
```

Если sale price меньше Einkaufspreis gemäß §24, Differenz-USt = 0, но экономический убыток должен быть виден.

### 12.3 Regelbesteuerung

При Regelbesteuerung не показывать поле `Differenz-USt`. Показывать:

- sale net;
- Ausgangs-USt;
- sale gross;
- purchase net;
- purchase USt;
- purchase gross;
- deductible input VAT amount как отдельное подтверждаемое поле;
- итоговую маржу.

Расчёт при ставке 20% и gross sale input:

```text
outputVat = saleGross * 20 / 120
saleNet = saleGross - outputVat
```

Экономическая стоимость покупки:

```text
economicPurchaseCost = purchaseGross - confirmedDeductibleInputVat
```

Маржа:

```text
finalVehicleMargin =
  saleNet
  - economicPurchaseCost
  - net-or-gross economic amount of linked expenses according to their deductible VAT flag
```

Не выводить Vorsteuer автоматически только из ставки: хранить отдельно подтверждённую сумму deductible input VAT.

### 12.4 План и факт

Для непроданного автомобиля считать прогноз по plannedSalePrice. После продажи все отчёты должны использовать actualSalePrice. В интерфейсе явно разделить:

- `Плановая маржа`;
- `Фактическая маржа`.

## 22. Миграция текущей MVP

Перед изменением схемы сделать backup и безопасную миграцию.

Предлагаемый mapping существующих статусов:

```text
Подготовка   -> IN_PREPARATION
В наличии   -> READY_FOR_SALE
Бронь       -> RESERVED
Продан      -> SOLD
```

Правила миграции:

1. Не конвертировать числовые суммы по валютному курсу. Считать, что текущие тестовые/введённые числа должны отображаться как EUR, если нет отдельного currency field.
2. Существующие расходы автомобиля перенести в новую Expense model и связать с vehicleId.
3. Завершённые Deal при наличии данных преобразовать в Sale.
4. Незавершённые Deal не удалять: сохранить как legacy до ручной проверки либо преобразовать очевидные test-drive события в Appointment, а действия — в Task.
5. Удалить Deal UI только после того, как ссылки и dashboard больше от него не зависят.
6. Не допускать orphan records.
7. Миграция должна быть повторяемой или иметь чёткую защиту от повторного запуска.
8. Добавить seed с австрийскими обезличенными тестовыми данными, не использовать реальные персональные данные.
